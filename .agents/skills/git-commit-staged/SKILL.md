# Skill: Git Commit Staged Changes

This skill helps you inspect currently staged changes in git, generate a high-quality commit message following the Conventional Commits specification, and execute the commit.

## Pre-requisites & Verification

1. **Verify Git Staging**:
   Run the helper script to verify that there are changes staged for commit:
   ```bash
   .agents/skills/git-commit-staged/scripts/commit_helper.sh
   ```
   - If the helper script exits with an error or reports that no changes are staged, print the list of modified/untracked files to the user and ask them to stage the files they want to commit using `git add`. Do not proceed until there are staged changes.

2. **Inspect Staged Changes**:
   Run the following command to review the staged diff:
   ```bash
   git diff --cached
   ```
   - If the diff is extremely large, run `git diff --cached --stat` first to see a summary of the changes, then selectively look at the diffs of the most critical files if needed.

## Commit Message Guidelines

Draft a commit message following the **Conventional Commits** specification:

### 1. Format
```
<type>(<scope>): <short description in imperative mood>

[optional body explaining why the change was made and what it does]

[optional footer(s)]
```

### 2. Types
- `feat`: A new feature
- `fix`: A bug fix
- `docs`: Documentation-only changes
- `style`: Formatting, missing semi-colons, white-space (no code logic changes)
- `refactor`: Code change that neither fixes a bug nor adds a feature
- `perf`: Code change that improves performance
- `test`: Adding or correcting tests
- `build`: Changes affecting build system, packaging, or external dependencies (e.g., npm, cargo, pip)
- `ci`: Changes to CI configuration files/scripts (e.g., GitHub Actions)
- `chore`: Maintenance tasks, updating dependencies, or other changes not affecting src or test files
- `revert`: Reverts a previous commit

### 3. Subject Line Rules
- Use the **imperative mood** (e.g., "add parser" instead of "added parser" or "adds parser").
- Do **not** capitalize the first letter.
- Do **not** end the subject line with a period.
- Keep the subject line under **50 characters** if possible (hard limit of 72 characters).

### 4. Body Rules
- Separate the body from the subject line with a blank line.
- Explain the **why** and **what** of the changes, rather than the **how** (unless how is unusual/complex).
- Wrap body lines at **72 characters**.

### 5. Footers & Breaking Changes
- If there are breaking changes, append `!` to the type/scope (e.g. `feat(api)!: change authentication endpoint`) AND/OR include `BREAKING CHANGE: <description>` in the footer.

## Commit Workflow

1. **Draft and Propose**:
   Draft the commit message based on the diff you reviewed. Present the proposed message clearly to the user, highlighting the type, scope, subject, and any body/footer details.
   - *Exception*: If the user's prompt explicitly requested automatic committing without asking (e.g., "commit the staged changes directly" or "commit with an autogenerated message"), you may proceed to step 2 directly.

2. **Execute Commit**:
   Commit the staged changes using:
   ```bash
   git commit -m "<subject>" -m "<body>"
   ```
   If there is no body, simply run:
   ```bash
   git commit -m "<subject>"
   ```

3. **Verify Commit**:
   Verify that the commit was successful:
   ```bash
   git status
   ```
   Or print the latest commit to confirm:
   ```bash
   git log -n 1 --oneline
   ```

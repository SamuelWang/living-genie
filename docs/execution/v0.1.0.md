# Execution plan — v0.1.0

Ordered task breakdown for implementing v0.1.0, sequencing the work described in
[../architecture.md](../architecture.md) against the acceptance criteria in
[../requirements/v0.1.0.md](../requirements/v0.1.0.md). This doc tracks *what to build and in what
order*, not design decisions — those live in architecture.md.

## 1. Repo scaffolding

1. [x] Create `web/` and `web-api/` per architecture.md's [repository layout](../architecture.md#repository-layout-proposed).
   - [x] Init `web-api/` as a Python project (`pyproject.toml` or `requirements.txt`).
   - [x] Init `web/` as a Node project (Vite + React + TS template).
   - [x] Add `.gitignore` entries for both (venv/`__pycache__`, `node_modules`, build output).

## 2. Backend foundation

1. [x] Init FastAPI project structure in `web-api/`.
   - [x] `app/main.py` with the FastAPI app instance.
   - [x] Settings/config module reading env vars (e.g. `DATABASE_URL`).
   - [x] DB session dependency (SQLAlchemy engine + sessionmaker, FastAPI dependency for request-scoped sessions).
2. [x] SQLAlchemy model for `diary_entries` per the [data model](../architecture.md#data-model).
   - [x] Declarative base + `DiaryEntry` model with typed columns (`id`, `title`, `content`, `entry_date`, `created_at`, `updated_at`).
3. [x] Alembic setup + initial migration.
   - [x] `alembic init`, wire `env.py` to the SQLAlchemy metadata and `DATABASE_URL`.
   - [x] Generate + review initial migration creating `diary_entries`.
4. [x] Pydantic request/response schemas.
   - [x] `DiaryEntryCreate` (title, content, entry_date).
   - [x] `DiaryEntryUpdate` (partial: title/content/entry_date).
   - [x] `DiaryEntryRead` (full entry incl. id/created_at/updated_at) and a lighter `DiaryEntrySummary` for list responses.

## 3. Backend CRUD + uploads

1. [x] `POST /diaries` — create.
   - [x] Reject missing title or entry_date (422).
   - [x] Default `entry_date` to today when omitted.
   - [x] Persist and return the created entry.
2. [x] `GET /diaries` — list.
   - [x] Query ordered by `entry_date` desc.
   - [x] Return summary fields (id, title, entry_date).
3. [x] `GET /diaries/{id}` — read.
   - [x] 404 when id doesn't exist.
   - [x] Return full entry (title, entry_date, content).
4. [x] `PUT /diaries/{id}` — update.
   - [x] 404 when id doesn't exist.
   - [x] Apply partial changes, refresh `updated_at`.
5. [x] `DELETE /diaries/{id}` — delete.
   - [x] 404 when id doesn't exist.
   - [x] Remove the row (no orphan-image cleanup required, per requirements.md).
6. [x] `POST /uploads/images` — upload.
   - [x] Accept multipart file, generate a unique filename, save under the uploads volume path.
   - [x] Return the public URL for the editor to embed.
   - [x] Mount the uploads directory as static files (e.g. under `/media`).

## 7. Frontend flows

Mapped to requirements/v0.1.0.md's functional requirements 1–6:

1. [ ] Create entry.
   - [ ] Create form: title input, entry-date picker (default today), editor for content.
   - [ ] Client-side validation blocking save without title/date.
   - [ ] On success, navigate to/refresh list so the new entry is immediately visible.
2. [ ] List entries.
   - [ ] List page fetching `GET /diaries`.
   - [ ] Render title + entry date per row, chronological order preserved from API.
3. [ ] View single entry.
   - [ ] Detail page fetching `GET /diaries/{id}`.
   - [ ] Render title, entry date, and markdown content.
4. [ ] Edit entry.
   - [ ] Edit form pre-filled from `GET /diaries/{id}`.
   - [ ] Save calls `PUT /diaries/{id}`; list re-sorts if entry_date changed.
5. [ ] Insert image.
   - [ ] Wire editor's upload button to `POST /uploads/images`.
   - [ ] Wire drag-and-drop into the editor to the same upload call.
   - [ ] Insert returned URL as an inline image in the content.
6. [ ] Delete entry.
   - [ ] Delete action opens a confirmation dialog.
   - [ ] On confirm, call `DELETE /diaries/{id}` and remove the entry from the list view.

# Execution plan — v0.1.0

Ordered task breakdown for implementing v0.1.0, sequencing the work described in
[../architecture.md](../architecture.md) against the acceptance criteria in
[../requirements/v0.1.0.md](../requirements/v0.1.0.md). This doc tracks *what to build and in what
order*, not design decisions — those live in architecture.md.

## 1. Repo scaffolding

1. [ ] Create `frontend/` and `backend/` per architecture.md's [repository layout](../architecture.md#repository-layout-proposed).
   - [ ] Init `backend/` as a Python project (`pyproject.toml` or `requirements.txt`).
   - [ ] Init `frontend/` as a Node project (Vite + React + TS template).
   - [ ] Add `.gitignore` entries for both (venv/`__pycache__`, `node_modules`, build output).

## 2. Backend foundation

1. [ ] Init FastAPI project structure in `backend/`.
   - [ ] `app/main.py` with the FastAPI app instance.
   - [ ] Settings/config module reading env vars (e.g. `DATABASE_URL`).
   - [ ] DB session dependency (SQLAlchemy engine + sessionmaker, FastAPI dependency for request-scoped sessions).
2. [ ] SQLAlchemy model for `diary_entries` per the [data model](../architecture.md#data-model).
   - [ ] Declarative base + `DiaryEntry` model with typed columns (`id`, `title`, `content`, `entry_date`, `created_at`, `updated_at`).
3. [ ] Alembic setup + initial migration.
   - [ ] `alembic init`, wire `env.py` to the SQLAlchemy metadata and `DATABASE_URL`.
   - [ ] Generate + review initial migration creating `diary_entries`.
4. [ ] Pydantic request/response schemas.
   - [ ] `DiaryEntryCreate` (title, content, entry_date).
   - [ ] `DiaryEntryUpdate` (partial: title/content/entry_date).
   - [ ] `DiaryEntryRead` (full entry incl. id/created_at/updated_at) and a lighter `DiaryEntrySummary` for list responses.

## 3. Backend CRUD + uploads

1. [ ] `POST /diaries` — create.
   - [ ] Reject missing title or entry_date (422).
   - [ ] Default `entry_date` to today when omitted.
   - [ ] Persist and return the created entry.
2. [ ] `GET /diaries` — list.
   - [ ] Query ordered by `entry_date` desc.
   - [ ] Return summary fields (id, title, entry_date).
3. [ ] `GET /diaries/{id}` — read.
   - [ ] 404 when id doesn't exist.
   - [ ] Return full entry (title, entry_date, content).
4. [ ] `PUT /diaries/{id}` — update.
   - [ ] 404 when id doesn't exist.
   - [ ] Apply partial changes, refresh `updated_at`.
5. [ ] `DELETE /diaries/{id}` — delete.
   - [ ] 404 when id doesn't exist.
   - [ ] Remove the row (no orphan-image cleanup required, per requirements.md).
6. [ ] `POST /uploads/images` — upload.
   - [ ] Accept multipart file, generate a unique filename, save under the uploads volume path.
   - [ ] Return the public URL for the editor to embed.
   - [ ] Mount the uploads directory as static files (e.g. under `/media`).

## 4. Backend tests

1. [ ] Unit tests for business logic.
   - [ ] Validation rules (title/entry_date required, defaulting behavior).
   - [ ] `updated_at` refresh logic on edit.
2. [ ] Integration tests against a real/test PostgreSQL instance.
   - [ ] Test DB fixture (setup/teardown, isolated from dev DB).
   - [ ] Full CRUD lifecycle test (create → list → get → update → delete).
   - [ ] Upload endpoint test (file saved, URL returned, static route serves it).

## 5. Frontend foundation

1. [ ] React + TypeScript scaffold in `frontend/`.
   - [ ] Routing setup (react-router) for list/detail/create/edit views.
   - [ ] Base project structure (`pages/`, `components/`, `api/`).
2. [ ] Tailwind CSS + shadcn/ui setup.
   - [ ] Install and configure Tailwind.
   - [ ] Init shadcn/ui, add base theme/components used across views (button, dialog, input).
3. [ ] Typed API client.
   - [ ] TS types mirroring the backend Pydantic schemas.
   - [ ] Fetch wrapper functions for each of the 6 endpoints (5 diary + 1 upload).

## 6. Frontend editor

1. [ ] Tiptap integration.
   - [ ] Install Tiptap + StarterKit (headings, lists incl. task lists, blockquote, code blocks, links).
   - [ ] Add GFM tables + strikethrough extension.
   - [ ] Add text-style/color extension.
   - [ ] Add image extension.
2. [ ] Markdown serialization via `tiptap-markdown`.
   - [ ] Editor content → markdown string on save.
   - [ ] Markdown string → editor content on load (existing entry edit).
   - [ ] Confirm non-markdown formatting (text color) round-trips as inline HTML.

## 7. Frontend flows

Mapped to requirements/v0.1.0.md's functional requirements 1–6:

1. [ ] Create entry.
   - [ ] Create form: title input, entry-date picker (default today), editor for content.
   - [ ] Client-side validation blocking save without title/date.
   - [ ] On success, navigate to/refresh list so the new entry is immediately visible.
2. [ ] List entries.
   - [ ] List page fetching `GET /diaries`.
   - [ ] Render title + entry date per row, chronological order preserved from API.
3. [ ] View single entry.
   - [ ] Detail page fetching `GET /diaries/{id}`.
   - [ ] Render title, entry date, and markdown content.
4. [ ] Edit entry.
   - [ ] Edit form pre-filled from `GET /diaries/{id}`.
   - [ ] Save calls `PUT /diaries/{id}`; list re-sorts if entry_date changed.
5. [ ] Insert image.
   - [ ] Wire editor's upload button to `POST /uploads/images`.
   - [ ] Wire drag-and-drop into the editor to the same upload call.
   - [ ] Insert returned URL as an inline image in the content.
6. [ ] Delete entry.
   - [ ] Delete action opens a confirmation dialog.
   - [ ] On confirm, call `DELETE /diaries/{id}` and remove the entry from the list view.

## 8. Frontend tests

1. [ ] Vitest + React Testing Library unit/component tests.
   - [ ] Form validation (create/edit) blocks missing title/date.
   - [ ] List renders entries in the order the API returns them.
   - [ ] Delete confirmation dialog gates the delete call.
2. [ ] Playwright e2e.
   - [ ] Full CRUD flow through the UI: create → appears in list → view → edit → delete.
   - [ ] Image insert flow: upload/drag-drop an image, verify it renders in the saved entry.

## 9. Containerization

1. [ ] `Dockerfile` for frontend.
   - [ ] Multi-stage build: install/build, then serve static output.
2. [ ] `Dockerfile` for backend.
   - [ ] Python base image, install deps, run via uvicorn.
3. [ ] Root `docker-compose.yml`.
   - [ ] Services: `frontend`, `backend`, `postgres`.
   - [ ] Named volume for Postgres data.
   - [ ] Named volume for uploaded images, mounted into `backend`.
4. [ ] Env var config.
   - [ ] `DATABASE_URL` wired from compose into the backend service.
   - [ ] `.env.example` documenting required vars.

## 10. Final acceptance walkthrough

1. [ ] `docker compose up` brings up frontend, backend, and database, ready to use.
   - [ ] All three containers report healthy/ready.
   - [ ] Migrations apply automatically or via a documented one-off command.
2. [ ] Through the UI: create, list, view, edit, and delete diary entries end-to-end.
   - [ ] Manual pass through all 6 functional requirements from requirements.md.
   - [ ] Confirm data survives a container restart (`docker compose down && up`).

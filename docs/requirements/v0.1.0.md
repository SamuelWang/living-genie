# Requirements — v0.1.0

## Overview

Living Genie is a personal living assistant web application that will eventually integrate
local LLM models (via Ollama) to help a user work with their own personal data — starting
with a diary. This document covers **v0.1.0 only**, per [../roadmap.md](../roadmap.md): standing
up the frontend/backend infrastructure, adding user registration and login, and delivering full
diary CRUD (create, edit, update, delete) scoped to the authenticated user. No AI/LLM
functionality is part of this phase — see [Non-goals](#non-goals--out-of-scope).

For the technical design supporting these requirements, see [../architecture.md](../architecture.md).

## Goals

- Stand up a working frontend and backend, containerized and runnable locally.
- Add user registration and login so diary data is private to each account.
- Deliver diary create/list/view/edit/delete as the first usable feature of the product.
- Support a multi-language UI, defaulting to Traditional Chinese.

## Non-goals / Out of scope

The following are explicitly **not** part of v0.1.0:

- RAG, chunking, or any embedding/retrieval pipeline over diary content (v0.2.0).
- Any feature backed by an LLM (Ollama or otherwise).
- Password reset / forgot-password flow.
- Email verification of the registration address.
- OAuth/social login.
- Role-based access control — every account has identical permissions over only its own data.
- Search or filtering beyond a plain chronological list of entries.
- Tags, mood, or other diary metadata beyond title/content/date.
- Additional UI languages beyond Traditional Chinese and English.
- Syncing the language preference to the user's account — it lives client-side only in v0.1.0.
- Translating or localizing user-authored diary content (titles/entries are never translated).

## Users

Living Genie v0.1.0 is a **multi-user, local-only** application: each person registers their own
account (email + password) and logs in; all diary data and uploaded images are scoped to the
authenticated account. There is no account management beyond registration/login (no password
reset, no profile editing, no admin roles).

## Functional requirements

### 1. Register an account

As a new user, I can create an account with an **email** and **password**.

**Acceptance criteria**
- Registering with an email already in use fails with a clear error; no password specifics are
  leaked either way.
- The password is never stored or returned in plaintext.
- Registering does not automatically log the user in — logging in is a separate step (requirement 2).

### 2. Log in

As a registered user, I can log in with my **email** and **password**.

**Acceptance criteria**
- Correct credentials start a session that grants access to that account's own diary data (see
  [architecture.md](../architecture.md) for the session mechanism).
- Incorrect credentials (wrong password or unregistered email) show one generic error that does
  not reveal whether the email is registered.
- The user can log out, ending the session.

### 3. Create a diary entry

As the user, I can create a new diary entry with:
- A **title**
- **Content**, written and formatted via a markdown WYSIWYG editor
- An **entry date**, defaulting to today but changeable by the user (e.g. to back-date an entry
  written after the fact)

**Acceptance criteria**
- A new entry cannot be saved without a title and an entry date.
- The entry is persisted and immediately visible in the entry list after creation.
- Creating an entry requires being logged in; the entry is owned by the authenticated account.

### 4. View the list of diary entries

As the user, I can see all my diary entries in a list, ordered by entry date (most recent first),
each showing at minimum its title and entry date.

**Acceptance criteria**
- The list reflects entry date ordering, not creation order (a back-dated entry appears in its
  correct chronological position).
- The list only ever includes the authenticated account's own entries.

### 5. View a single diary entry

As the user, I can open an entry from the list and see its title, entry date, and content
rendered from markdown.

**Acceptance criteria**
- Viewing requires being logged in as the entry's owner; another account's entry behaves
  identically to a nonexistent one (not found, not a permissions error).

### 6. Edit an existing diary entry

As the user, I can edit an existing entry's title, content, and/or entry date.

**Acceptance criteria**
- Saving an edit updates the entry in place and refreshes its `updated_at` record timestamp.
- Editing the entry date moves the entry to its new position in the chronological list.
- Only the entry's owning account can edit it; another account's edit attempt behaves identically
  to editing a nonexistent entry.

### 7. Insert an image into a diary entry

As the user, I can insert an image into an entry's content via the editor (upload or
drag-and-drop). The image is embedded inline and persists with the entry.

**Acceptance criteria**
- An uploaded image appears inline when viewing or editing the entry.
- Deleting the entry does not need to guarantee cleanup of its uploaded images in v0.1.0 — no
  orphan-file cleanup requirement yet.
- Uploading requires being logged in.

### 8. Delete a diary entry

As the user, I can delete an entry, with a confirmation step to prevent accidental loss.

**Acceptance criteria**
- Once confirmed, the entry is removed and no longer appears in the list.
- Only the entry's owning account can delete it; another account's delete attempt behaves
  identically to deleting a nonexistent entry.

### 9. Switch UI language

As a user, I can view the app in Traditional Chinese or English and switch between them.

**Acceptance criteria**
- On first visit, the UI uses the browser's detected language if it's supported (`zh-Hant` or
  `en`); otherwise it defaults to Traditional Chinese.
- A visible control lets the user switch language at any time; the change applies immediately,
  with no reload required.
- The chosen language persists across sessions via the browser (e.g. `localStorage`), not synced
  to the account.
- All UI chrome (nav, forms, buttons, validation/error messages, empty states) is translated in
  both locales; diary content itself is never translated or altered.

## Data requirements

Each user account has:

| Field             | Description                                              |
|-------------------|-------------------------------------------------------------|
| `id`              | Unique identifier                                            |
| `email`           | Unique login identifier                                      |
| `hashed_password` | Password stored as a one-way hash, never plaintext           |
| `created_at`      | System-managed timestamp of account creation                 |

Each diary entry has:

| Field        | Description                                                  |
|--------------|----------------------------------------------------------------|
| `id`         | Unique identifier                                              |
| `user_id`    | The owning account's `id`; system-managed, not user-editable   |
| `title`      | Entry title                                                     |
| `content`    | Entry body, stored as markdown text. May include embedded images referenced by URL (standard markdown image syntax) — images are not a separate field or entity. |
| `entry_date` | User-selectable date the entry is about; defaults to today, editable |
| `created_at` | System-managed timestamp of record creation, not user-editable  |
| `updated_at` | System-managed timestamp of last modification, not user-editable |

## Non-functional requirements

- The full application (frontend, backend, database) runs locally via Docker.
- Diary data is persisted in PostgreSQL and survives container restarts.
- The backend exposes a REST API consumed by the frontend.
- No specific performance/scale targets — the app serves a single local user.
- Both frontend and backend must have unit tests and integration tests covering the diary CRUD
  and auth (registration/login) paths. No specific coverage percentage is required for v0.1.0.

## Acceptance criteria for v0.1.0 completion

- Running `docker compose up` brings up the frontend, backend, and database, ready to use.
- Through the UI, the user can register a new account and log in.
- Through the UI, the user can create, list, view, edit, and delete diary entries end-to-end.
- Through the UI, the user can view and switch between Traditional Chinese (default) and English.

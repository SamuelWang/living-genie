# Requirements — v0.1.0

## Overview

Living Genie is a personal living assistant web application that will eventually integrate
local LLM models (via Ollama) to help a user work with their own personal data — starting
with a diary. This document covers **v0.1.0 only**, per [../roadmap.md](../roadmap.md): standing
up the frontend/backend infrastructure and delivering full diary CRUD (create, edit, update,
delete). No AI/LLM functionality is part of this phase — see [Non-goals](#non-goals--out-of-scope).

For the technical design supporting these requirements, see [../architecture.md](../architecture.md).

## Goals

- Stand up a working frontend and backend, containerized and runnable locally.
- Deliver diary create/list/view/edit/delete as the first usable feature of the product.

## Non-goals / Out of scope

The following are explicitly **not** part of v0.1.0:

- RAG, chunking, or any embedding/retrieval pipeline over diary content (v0.2.0).
- Any feature backed by an LLM (Ollama or otherwise).
- Multi-user support or user accounts.
- Authentication / authorization of any kind.
- Search or filtering beyond a plain chronological list of entries.
- Tags, mood, or other diary metadata beyond title/content/date.

## Users

Living Genie v0.1.0 is a **single-user, local-only** application. It runs on the user's own
machine with no login — there is exactly one implicit user and no account management.

## Functional requirements

### 1. Create a diary entry

As the user, I can create a new diary entry with:
- A **title**
- **Content**, written and formatted via a markdown WYSIWYG editor
- An **entry date**, defaulting to today but changeable by the user (e.g. to back-date an entry
  written after the fact)

**Acceptance criteria**
- A new entry cannot be saved without a title and an entry date.
- The entry is persisted and immediately visible in the entry list after creation.

### 2. View the list of diary entries

As the user, I can see all my diary entries in a list, ordered by entry date (most recent first),
each showing at minimum its title and entry date.

**Acceptance criteria**
- The list reflects entry date ordering, not creation order (a back-dated entry appears in its
  correct chronological position).

### 3. View a single diary entry

As the user, I can open an entry from the list and see its title, entry date, and content
rendered from markdown.

### 4. Edit an existing diary entry

As the user, I can edit an existing entry's title, content, and/or entry date.

**Acceptance criteria**
- Saving an edit updates the entry in place and refreshes its `updated_at` record timestamp.
- Editing the entry date moves the entry to its new position in the chronological list.

### 5. Insert an image into a diary entry

As the user, I can insert an image into an entry's content via the editor (upload or
drag-and-drop). The image is embedded inline and persists with the entry.

**Acceptance criteria**
- An uploaded image appears inline when viewing or editing the entry.
- Deleting the entry does not need to guarantee cleanup of its uploaded images in v0.1.0 — no
  orphan-file cleanup requirement yet.

### 6. Delete a diary entry

As the user, I can delete an entry, with a confirmation step to prevent accidental loss.

**Acceptance criteria**
- Once confirmed, the entry is removed and no longer appears in the list.

## Data requirements

Each diary entry has:

| Field        | Description                                                  |
|--------------|----------------------------------------------------------------|
| `id`         | Unique identifier                                              |
| `title`      | Entry title                                                     |
| `content`    | Entry body, stored as markdown text. May include embedded images referenced by URL (standard markdown image syntax) — images are not a separate field or entity. |
| `entry_date` | User-selectable date the entry is about; defaults to today, editable |
| `created_at` | System-managed timestamp of record creation, not user-editable  |
| `updated_at` | System-managed timestamp of last modification, not user-editable |

## Non-functional requirements

- The full application (frontend, backend, database) runs locally via Docker.
- Diary data is persisted in PostgreSQL and survives container restarts.
- The backend exposes a REST API consumed by the frontend.
- No specific performance/scale targets — the app serves a single local user.
- Both frontend and backend must have unit tests and integration tests covering the diary CRUD
  paths. No specific coverage percentage is required for v0.1.0.

## Acceptance criteria for v0.1.0 completion

- Running `docker compose up` brings up the frontend, backend, and database, ready to use.
- Through the UI, the user can create, list, view, edit, and delete diary entries end-to-end.
